Sì, ci sono altri freezer UI oltre alle API Betfair. Quelli più comuni in app Tkinter/ttkbootstrap come Pickfair sono questi:
1) Creazione/aggiornamento massivo di widget (Treeview / Frame / Label)
Se quando clicchi “Dashboard” o “Nazione” fai:
distruggi e ricrei 200–2000 widget
inserisci 500 righe in Treeview in un colpo 👉 Tkinter freeza anche senza rete.
Fix veloce
inserisci dati “a blocchi” con root.after() (chunking)
oppure aggiorni solo le righe cambiate
Esempio chunk:
Copia codice
Python
def fill_tree_chunked(items, i=0, chunk=50):
    end = min(i+chunk, len(items))
    for x in items[i:end]:
        tree.insert("", "end", values=x)
    if end < len(items):
        root.after(1, lambda: fill_tree_chunked(items, end, chunk))
2) Loop “while” o sleep nel main thread
Qualsiasi cosa tipo:
Copia codice
Python
while something:
    time.sleep(1)
o anche solo:
Copia codice
Python
time.sleep(2)
nel main thread = UI morta.
Fix
usare root.after() invece di loop/sleep
3) Funzioni troppo lunghe chiamate da click (anche senza I/O)
Esempio:
parsing enorme
calcoli dutching pesanti
calcolo P&L su tanti mercati
matching team names con fuzzy su 1000 righe
Fix
sposta in background thread
oppure fai “yield” periodico:
Copia codice
Python
root.update_idletasks()
(ma meglio background)
4) Deadlock / lock condivisi tra thread
Se hai:
threading.Lock()
queue.get() senza timeout
join() su thread in UI
Esempio killer:
Copia codice
Python
thread.join()  # MAI nel main thread
Oppure:
Copia codice
Python
q.get()  # senza timeout
Fix
q.get(timeout=...)
mai join nel main thread
5) Logging troppo aggressivo sul thread UI
Se logghi 200 righe al secondo e scrivi su file lento, può laggare.
Fix
usa QueueHandler per logging async (log in thread separato)
oppure riduci DEBUG in release
6) Messagebox chiamate da thread background
Questo spesso non freeza, ma può fare comportamenti strani/crash.
Regola
messagebox.showinfo() SOLO nel main thread:
Copia codice
Python
self.root.after(0, lambda: messagebox.showinfo(...))
7) Immagini / icone / ridimensionamenti continui
Se fai refresh continuo di:
immagini (PIL)
canvas redraw
icone + resize ogni tick
👉 può freezare o diventare laggy.
Fix
cache immagini
non ricreare PhotoImage ogni volta
8) Troppi .after() che si accumulano (timer duplicati)
Classico bug: ogni click crea un nuovo timer, e dopo 2 minuti hai 500 timer attivi.
Fix
salva l’id del timer e cancella prima:
Copia codice
Python
if self._job:
    self.root.after_cancel(self._job)
self._job = self.root.after(1000, self.tick)
9) DB SQLite nel main thread (soprattutto con VACUUM / backup)
Se fai:
query grosse
backup
vacuum nel click UI → freeze.
Fix
DB in thread separato
10) Stream / Telethon callback che tocca UI direttamente
Telethon e stream Betfair possono chiamare callback in thread loro. Se dentro fai update widget → freeze/crash random.
Fix
ogni update UI deve passare da:
Copia codice
Python
self.root.after(0, lambda: update_ui(...))
Quelli più probabili nel tuo caso (Pickfair)
📌 i 3 top che ti bloccano “appena clicco”:
API chiamata in main thread (già visto)
Treeview / widget massivi (dashboard / eventi)
timer duplicati after() (refresh che si moltiplica)
Se vuoi blindare TUTTO con 1 sola cosa
Metti questo controllo:
🔥 “Qualsiasi callback UI lenta = log + thread dump”
Copia codice
Python
def ui_guard_dump(name: str, warn_ms=200, dump_ms=2000):
    def deco(fn):
        def wrapper(*args, **kwargs):
            t0 = time.perf_counter()
            try:
                return fn(*args, **kwargs)
            finally:
                dt = (time.perf_counter() - t0) * 1000
                if dt > warn_ms:
                    logger.warning(f"[UI-GUARD] {name} took {dt:.0f}ms")
                if dt > dump_ms:
                    logger.error(f"[UI-GUARD] FREEZE {name} took {dt:.0f}ms -> thread dump")
                    import faulthandler, sys
                    faulthandler.dump_traceback(file=sys.stderr, all_threads=True)
        return wrapper
    return deco
Poi lo metti su tutti i bottoni principali.
