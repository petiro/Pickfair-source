❌ PERCHÉ ERA UN BUG REALE
if not self.telegram_listener.running:
    logging.warning("Copy Trading: Telegram listener not running...")
    return


Questo controllo è concettualmente sbagliato perché:

telegram_listener serve a ricevere eventi (incoming)

send_message() usa un client di invio dedicato

I due flussi sono indipendenti

👉 Stavi bloccando l’output basandoti sullo stato dell’input

✅ COSA FARE (SOLUZIONE CORRETTA)
🔧 1️⃣ Rimuovere il return bloccante (come hai già intuito)

Versione corretta:

if not self.telegram_listener.running:
    logging.warning(
        "Copy Trading: Telegram listener not running, "
        "but broadcast is allowed (send client active)"
    )
# NESSUN return


Oppure ancora meglio: rimuovere del tutto il check.

🔧 2️⃣ Se vuoi un controllo corretto, verifica il SEND client

📌 Mai usare lo stato del listener per decidere l’invio

🧠 REGOLA ARCHITETTURALE (IMPORTANTE)

Receive path ≠ Send path

Componente	Serve per
Listener	Ricevere messaggi

Copy Trading	Evento interno

Se uno cade → l’altro non deve cadere.
